Indie QA automation without PDDL or binary magic

Bridge-Maker перетворює маленький семантичний контракт гри на автоматизований QA/RL тестер.

Розробник не пише PDDL, не налаштовує Ghidra і не вчить Ray/RLlib. Він позначає важливі стани, дії та оракули через прості декоратори. Bridge-Maker експортує контракт, запускає середовище і формує звіт з доказами багів.

Почати Українська презентація

1. Annotate

Позначте існуючі getters, command handlers та invariant checks.

2. Export

Отримайте `state_map.json`, `action_map.json`, `oracle_map.json` і `trace.jsonl`.

3. Report

Згенеруйте HTML/JSON звіт з oracle hits, action coverage і станами помилки.

Важлива зміна стратегії: Bridge-Maker більше не обіцяє магічно зрозуміти будь-який `.exe`. Сильна частина системи починається тоді, коли є мінімальний контракт: що таке HP, де позиція, які дії легальні, які стани є багами.

Швидкий старт

python -m bridge_maker --help
python -m bridge_maker demo --out runs/grant_demo
python -m bridge_maker smoke --adapter examples/buggy_roguelike.py --steps 20

Demo запускає повний MVP loop: suggestions → export → generated SDK env → report. Перевірено в репозиторії: CLI працює, unit tests проходять, demo створює `report.html`.

SDK decorators

Розробник не має писати нові системи тільки для Bridge-Maker. Він декорує вже існуючі функції або тонкий adapter навколо рушія/моду/реверс-проєкту. 

from bridge_maker import bm

@bm.hp(bounds=(0, 100))
def hp():
    return player.hp

@bm.position(x="x", y="y", bounds=(0, 100))
def position():
    return player.x, player.y

@bm.move("right", key="d")
def move_right():
    player.move_right()

@bm.oracle("out_of_bounds", severity="bug")
def out_of_bounds(state):
    return state.x < 0 or state.x > 99
Принцип: Bridge-Maker не змушує команду вивчати RL. Він просить дати мінімальну семантику, яку розробник і так знає.

Артефакти

Contract files

  • state_map.json - observation schema
  • action_map.json - legal action surface
  • oracle_map.json - bug/invariant checks
  • contract.json - manifest

Evidence files

  • trace.jsonl - action/state/oracle frames
  • report.json - machine-readable summary
  • report.html - shareable QA report
  • sdk_env_generated.py - generated env wrapper

Що вже готово

  • Python decorator SDK: `@bm.hp`, `@bm.position`, `@bm.action`, `@bm.oracle`, `@bm.reset`, `@bm.snapshot`.
  • Adapter loader і runtime registry.
  • Contract export у `state_map`, `action_map`, `oracle_map`, `trace`.
  • SDK Gymnasium environment для smoke loops.
  • AST CodeScout v0 для annotation suggestions.
  • JSON/HTML bug report з oracle evidence.
  • Buggy roguelike demo з реальним oracle hit.
  • Noita WebSocket adapter readiness notes і fake-client test.

Grey-box CE/Ghidra, Ray/RLlib і dashboard існують у репозиторії як дослідницький/наступний шар, але основний презентований MVP зараз - контракт, env і bug report без складного setup.

Roadmap

  1. MVP: SDK annotations → contract → trace → report. Status: ready.
  2. Adapter proof: live Noita/NoitaRL adapter over known reverse project.
  3. Agent assist: CodeScout/TraceScout пропонують missing annotations.
  4. Training: contract-backed Gym env feeds Ray/RLlib + curiosity.
  5. Overnight UX: one command returns bug report, replay traces, and trainer artifacts.

Як презентувати

Позиціонування має бути чесним: це не "бот, який сам зрозуміє будь-який бінарник". Це framework, де розробник дає мінімальний контракт, а агенти і RL шар роблять важку QA-роботу.

Bridge-Maker is the missing bridge between game semantics and automated QA: a few decorators in, reproducible bug evidence out.

Для demo показуйте `examples/buggy_roguelike.py`, команду `python -m bridge_maker demo`, `state_map.json` і фінальний `report.html`.
